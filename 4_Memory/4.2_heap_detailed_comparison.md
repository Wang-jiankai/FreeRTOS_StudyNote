# 4.2 对比详解：碎片 / 确定性 / 开销

## 碎片化问题

### 什么是内存碎片

碎片 = 分配和释放过程中产生的**无法使用的小空闲块**。

```
[IMAGE_PROMPT: Memory fragmentation diagram. Two scenarios:
Left: "无碎片" memory blocks showing clean allocation pattern with one large free block at end.
Right: "碎片化" memory blocks showing scattered small free blocks between allocated blocks, with total free space large but none contiguous enough for a new allocation.
Show "碎片 = 空闲内存总和够，但无法满足分配请求" label in Chinese. Highlight the problem with fragmented memory.}
```

### heap_2 碎片示例

```c
// 场景：多次分配相同大小对象
void task(void *p) {
    // 第一次运行
    void *p1 = pvPortMalloc(100);  // 分配
    vPortFree(p1);                  // 释放

    // 第二次运行
    void *p2 = pvPortMalloc(100);  // 分配
    vPortFree(p2);

    // N次后...
    // 内存变成：[100] [100-F] [100] [100-F] [100-F]...
    // 每个100字节的洞都是碎片
}
```

### heap_4 如何减少碎片

heap_4 在释放时**合并相邻空闲块**：

```c
// heap_4.c 中的合并逻辑
void vPortFree(void *pv) {
    BlockLink_t *pxBlockToFree = ...;

    /* 合并前面的空闲块 */
    if (pxBlockToFree->xBlockSize < configTOTAL_HEAP_SIZE) {
        pxBlockToFree = (BlockLink_t *)((uint8_t *)pxBlockToFree + pxBlockToFree->xBlockSize);
    }

    /* 合并后面的空闲块 */
    if (... 相邻 ...) {
        /* 合并 */
    }
}
```

## 确定性分析

### 什么是确定性

确定性 = 分配/释放的**时间复杂度是固定的**，不取决于内存使用状态。

| 类型 | 分配时间 | 说明 |
|------|---------|------|
| **确定性** | O(1) | 固定时间，无论内存状态 |
| **非确定性** | O(n) | 时间取决于空闲块数量 |

### 为什么 heap_1 是确定性的

heap_1 只是顺序分配，不需要搜索空闲块：

```c
// heap_1.c
void *pvPortMalloc(size_t xWantedSize) {
    if (xWantedSize > xNextFreeByte) {
        return NULL;  // 分配失败
    }

    void *ptr = &ucHeap[xNextFreeByte];
    xNextFreeByte += xWantedSize;
    return ptr;  // 时间固定
}
```

### 为什么 heap_4 不是确定性的

heap_4 需要遍历空闲链表找第一个够大的块：

```c
// heap_4.c
void *pvPortMalloc(size_t xWantedSize) {
    BlockLink_t *pxBlock = &xStart;
    BlockLink_t *pxPreviousBlock = &xStart;

    /* 遍历链表，找第一个够大的空闲块 */
    while (pxBlock->xBlockSize < xWantedSize) {  // ← 最坏 O(n)
        pxPreviousBlock = pxBlock;
        pxBlock = pxBlock->pxNextFreeBlock;
    }
    // ...
}
```

**时间取决于空闲块数量（取决于内存使用模式）**

## 内存开销

### 每种策略的额外开销

| 策略 | 头部长度 | 说明 |
|------|---------|------|
| heap_1 | 0 | 无头部 |
| heap_2 | 8 字节/块 | `BlockLink_t` 大小 |
| heap_3 | 0 | C 库管理 |
| heap_4 | 8 字节/块 | `BlockLink_t` 大小 |
| heap_5 | 8 字节/块 | 同 heap_4 |

### BlockLink_t 结构

```c
typedef struct A_BLOCK_LINK {
    struct A_BLOCK_LINK *pxNextFreeBlock;  // 下一个空闲块
    size_t xBlockSize;                       // 块大小（含头部）
} BlockLink_t;
```

### 实际开销计算

```
场景：创建 10 个任务，每个任务栈 256 字（1024 字节）

heap_2 / heap_4 开销：
- 每个块头部 8 字节
- 总开销 = 10 × 8 = 80 字节

heap 空间需求：
- 数据：10 × 1024 = 10240 字节
- 头部：80 字节
- 总计：≈ 10320 字节
```

## 性能对比

### 分配时间复杂度

| 操作 | heap_1 | heap_2 | heap_3 | heap_4 | heap_5 |
|------|--------|--------|--------|--------|--------|
| malloc | O(1) | O(n) | O(1)* | O(n) | O(n) |
| free | — | O(n) | O(1)* | O(n) | O(n) |
| 合并相邻 | — | ❌ | ✅ | ✅ | ✅ |

*C 库实现决定

### 内存利用率

| 策略 | 利用率 | 说明 |
|------|--------|------|
| heap_1 | 低 | 只分配不释放 |
| heap_2 | 中 | 不合并，有碎片 |
| heap_3 | 高 | C 库优化 |
| heap_4 | 高 | 合并效果好 |
| heap_5 | 高 | 同 heap_4 |

## 实际选型建议

### 选 heap_1 的场景

```c
// FreeRTOSConfig.h
#define configAPPLICATION_ALLOCATED_HEAP  1  // 用户提供堆内存

// 适合：简单的静态系统，任务创建后不删除
```

### 选 heap_4 的场景 ✅ 推荐

```c
// 适合：大多数嵌入式场景
// 任务会创建/删除，需要动态内存
```

### 选 heap_5 的场景

```c
// 适合：多块不连续内存的芯片
// 如：STM32F4 + 外扩 SRAM
vPortDefineHeapRegions(xHeapRegions);
```

---

## 本节小结

- **碎片**：heap_2 不合并会产生碎片，heap_4/5 会合并相邻空闲块
- **确定性**：heap_1 是 O(1)，其他取决于搜索时间
- **开销**：heap_2/4/5 每个块有 8 字节头部开销
- **实际选择**：大多数场景用 **heap_4**，简单静态系统用 **heap_1**
