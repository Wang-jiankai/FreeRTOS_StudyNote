# 6.4 踩坑实录：延时精度 / 空转功耗

## 坑 1：vTaskDelay 精度不确定

**现象**：
```c
vTaskDelay(pdMS_TO_TICKS(10));  // 期望 10ms，实际可能是 12ms+
```

**原因**：
- Tick 中断是周期性采样，不是精确延时
- 任务可能刚好错过最近一次 Tick
- 唤醒后还要等调度器调度

```
[IMAGE_PROMPT: Tick timing accuracy diagram. Show timeline with ticks at 1ms intervals. A task calls vTaskDelay(10ms) at T=2.3ms but doesn't actually wake until T=13ms (after the tick at T=12). Label "期望延时" and "实际延时". Show the jitter/error. Use Chinese labels.}
```

**解决**：
```c
// ✅ 使用 vTaskDelayUntil 获取确定性延时
TickType_t xWakeTime;
xWakeTime = xTaskGetTickCount();
for (;;) {
    vTaskDelayUntil(&xWakeTime, pdMS_TO_TICKS(10));  // 精确 10ms 周期
    process周期性任务();
}

// ✅ 高精度场景用硬件定时器 + 延时补偿
```

## 坑 2：vTaskDelay 在中断中不能用

**现象**：
```c
void TIM_IRQHandler(void) {
    vTaskDelay(pdMS_TO_TICKS(10));  // ❌ 硬错！
}
```

**原因**：
- `vTaskDelay` 会将任务从 Ready 链表移到 Delayed 链表
- 这涉及链表操作，不是中断安全函数

**解决**：
```c
// ✅ 中断中用硬件定时器延时
void TIM_IRQHandler(void) {
    // 简单延时用硬件定时器标志位
    // 或设置标志位，在任务中延时处理
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    xSemaphoreGiveFromISR(xSem, &xHigherPriorityTaskWoken);
}
```

## 坑 3：空转时 CPU 占用 100%

**现象**：
```c
void task(void *p) {
    while (1) {
        if (data_ready) {
            process();
        }
    }
}
```

**原因**：
- 任务无阻塞地跑满时间片
- 其他低优先级任务无法运行

**解决**：
```c
// ✅ 加延时让出 CPU
void task(void *p) {
    while (1) {
        if (data_ready) {
            process();
        }
        vTaskDelay(pdMS_TO_TICKS(1));  // 让出 CPU
    }
}

// ✅ 或用信号量阻塞
void task(void *p) {
    while (1) {
        xSemaphoreTake(xSem, portMAX_DELAY);  // 等待数据
        process();
    }
}
```

## 坑 4：Tick 中断太频繁导致功耗高

**现象**：
- 产品待机功耗超标
- 电池使用时间比预期短

**原因**：
- `configTICK_RATE_HZ = 1000`（1ms Tick）时，CPU 无法进入深度睡眠
- 每次 Tick 都唤醒 CPU，增加功耗

**解决**：

```c
// FreeRTOSConfig.h - 根据场景选 Tick 频率
#define configTICK_RATE_HZ    (100)  // 10ms 周期，功耗更低

// 或使用动态 Tick：在空闲时停止 Tick
#define configUSE_TICKLESS_IDLE    1  // 启用 tickless idle

// 当没有任务运行时，调度器停止 Tick
// 仅在下一个定时器到期或中断时唤醒
```

### Tickless 模式原理

```c
// 空闲任务中
void prvIdleTask(void) {
    while (1) {
        // 没有其他任务运行，停止 Tick
        if (listIS_EMPTY(&xPendingReadyList) == pdTRUE) {
            portSUPPRESS_TICKS_AND_SLEEP(xExpectedIdleTime);
            // CPU 进入深度睡眠，直到下一个事件唤醒
        }
    }
}
```

## 坑 5：定时器回调中操作共享数据

**现象**：
- 定时器回调和任务同时访问变量，数据不一致

**原因**：
- 定时器回调在 daemon 任务中执行
- daemon 可能和其他任务同时访问共享数据

**解决**：
```c
// ❌ 危险
void timer_callback(TimerHandle_t xTimer) {
    shared_variable++;  // 可能和其他任务冲突
}

// ✅ 用互斥量保护
void timer_callback(TimerHandle_t xTimer) {
    xSemaphoreTake(xMutex, portMAX_DELAY);
    shared_variable++;
    xSemaphoreGive(xMutex);
}

// ✅ 或使用 taskENTER_CRITICAL（注意不能在 daemon 阻塞）
```

## 本节小结

| 坑 | 原因 | 解决 |
|----|------|------|
| 延时精度差 | Tick 采样导致抖动 | 用 vTaskDelayUntil |
| 中断中延时 | vTaskDelay 非中断安全 | 用硬件定时器 |
| CPU 空转 100% | 任务无阻塞循环 | 加延时或信号量 |
| 功耗高 | Tick 太频繁 | 降低 Tick 频率或 tickless |
| 定时器数据冲突 | 回调和任务并发 | 互斥量保护 |
