# 7.4 踩坑实录：中断里不能 delay / 过长临界段

## 坑 1：中断中调用会阻塞的函数

**现象**：
```c
void UART_IRQHandler(void) {
    // ❌ 这些都会出问题
    vTaskDelay(10);                    // 硬错：vTaskDelay 不能在中断中用
    xQueueSend(xQueue, &data, portMAX_DELAY);  // ❌ 可能死锁
    xSemaphoreTake(xSem, portMAX_DELAY);      // ❌ 可能死锁
}
```

**原因**：
- `vTaskDelay` 会将当前任务挂起，触发调度
- 中断上下文不能切换任务
- `portMAX_DELAY` 在中断中意味着无限等待，但中断不能阻塞

**解决**：
```c
void UART_IRQHandler(void) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;

    // ✅ 用 FromISR 版本，超时用 0
    xQueueSendFromISR(xQueue, &data, &xHigherPriorityTaskWoken);

    // ✅ 如果队列满就丢弃，不阻塞
    xQueueSendFromISR(xQueue, &data, NULL);  // 第三个参数传 NULL

    if (xHigherPriorityTaskWoken) {
        portYIELD_FROM_ISR();
    }
}
```

## 坑 2：中断优先级超过 configMAX_SYSCALL_INTERRUPT_PRIORITY

**现象**：
```c
// 配置
#define configMAX_SYSCALL_INTERRUPT_PRIORITY  5

// 优先级 3 < 5，不能调 FromISR
NVIC_SetPriority(UART_IRQn, 3);

// 但代码中调用了
void UART_IRQHandler(void) {
    xQueueSendFromISR(xQueue, &data, NULL);  // ❌ 危险！
}
```

**后果**：
- 中断优先级太高，临界段关不掉它
- 调用 FreeRTOS API 时，可能打断内核数据结构操作
- 导致数据竞争、内存错误

**解决**：
```c
// ✅ 优先级必须 > 5
NVIC_SetPriority(UART_IRQn, 6);

// ✅ 或者把不紧急的中断改成低优先级
NVIC_SetPriority(不紧急中断, 7);
```

## 坑 3：临界段时间过长

**现象**：
```c
void bad_task(void *p) {
    taskENTER_CRITICAL();

    // 处理 1000 个数据
    for (int i = 0; i < 1000; i++) {
        process(data[i]);
    }

    taskEXIT_CRITICAL();  // 等了很久才打开中断
}
```

**后果**：
- 中断响应延迟（最多可达临界段执行时间）
- 串口接收丢字节
- 定时器 tick 丢失

```

- Critical section exits at T=100ms

```

**解决**：
```c
// ✅ 分批处理，每批后退出临界段
void good_task(void *p) {
    for (int i = 0; i < 1000; i++) {
        taskENTER_CRITICAL();
        process_one(data[i]);
        taskEXIT_CRITICAL();

        // 让中断有机会处理
        taskYIELD();
    }
}

// ✅ 或禁用调度器代替关中断
void alternative_task(void *p) {
    vTaskSuspendAll();  // 只禁止调度，不关中断
    process_all_data();
    xTaskResumeAll();   // 恢复调度
}
```

## 坑 4：中断中访问 FreeRTOS 对象后对象被删除

**现象**：
```c
// 任务中删除了队列
void delete_task(void *p) {
    vTaskDelay(pdMS_TO_TICKS(1000));
    xQueueDelete(xQueue);  // 删除了队列
}

// 但中断还在往队列发数据
void UART_IRQHandler(void) {
    xQueueSendFromISR(xQueue, &data, NULL);  // ❌ xQueue 已无效！
}
```

**后果**：
- 内存访问错误
- 硬错（HardFault）

**解决**：
```c
// ✅ 删除前确保没有中断在使用
void safe_delete_task(void *p) {
    // 禁用相关中断
    NVIC_DisableIRQ(UART_IRQn);

    // 等待，确认中断退出
    taskENTER_CRITICAL();
    xQueueDelete(xQueue);
    xQueue = NULL;
    taskEXIT_CRITICAL();

    NVIC_EnableIRQ(UART_IRQn);
}

// ✅ 或使用引用计数
```

## 坑 5：临界段嵌套忘记配对

**现象**：
```c
void broken_function(void) {
    taskENTER_CRITICAL();  // 第 1 次

    if (error) {
        return;  // ❌ 提前返回，没退出临界段！
    }

    taskENTER_CRITICAL();  // 第 2 次
    // ...
    taskEXIT_CRITICAL();  // 只退出第 2 次
    taskEXIT_CRITICAL();  // 第 1 次
}
```

**后果**：
- 中断被永久关闭
- 系统无响应

**解决**：
```c
// ✅ 用 goto 统一出口
void correct_function(void) {
    taskENTER_CRITICAL();

    if (error) {
        goto cleanup;
    }

    taskENTER_CRITICAL();
    if (another_error) {
        goto cleanup2;
    }

cleanup2:
    taskEXIT_CRITICAL();
cleanup:
    taskEXIT_CRITICAL();
}
```

## 坑 6：在中断中做浮点运算

**现象**：
```c
void FP_IRQHandler(void) {
    float result = compute();  // ❌ 浮点运算很慢
}
```

**原因**：
- 浮点运算耗时，可能导致中断响应不及时
- M4/M7 的 FPU 上下文切换有开销

**解决**：
```c
// ✅ 在任务中做浮点运算，中断只设置标志
void FP_IRQHandler(void) {
    data_ready = 1;  // 只设置标志
}

// 任务中处理
void processing_task(void *p) {
    while (1) {
        if (data_ready) {
            data_ready = 0;
            float result = compute();
        }
        vTaskDelay(pdMS_TO_TICKS(1));
    }
}
```

## 中断设计最佳实践

| 原则 | 说明 |
|------|------|
| 快进快出 | 中断处理要短，中断标志置 1 或发信号量 |
| 非阻塞 | 不用 `portMAX_DELAY`，用 0 或有限超时 |
| 优先级正确 | > `configMAX_SYSCALL_INTERRUPT_PRIORITY` 才能调 API |
| 数据要安全 | 确保访问的对象不会被删除 |
| 临界段要短 | 毫秒级操作，最多几十毫秒 |

## 本节小结

| 坑 | 后果 | 解决 |
|----|------|------|
| 中断中阻塞 | 死锁、系统错乱 | 用 FromISR 版本 |
| 优先级超限 | 数据竞争 | 优先级 > 5 |
| 临界段太长 | 中断延迟、丢数据 | 分批、减少操作 |
| 对象删除后中断仍访问 | HardFault | 同步删除 + 禁用中断 |
| 嵌套不配对 | 中断永久关闭 | goto 统一出口 |
| 中断浮点运算 | 响应延迟 | 中断设标志，任务运算 |
