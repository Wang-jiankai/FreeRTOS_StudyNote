# 1.4 调参指南：栈大小怎么算、优先级怎么设

## 栈大小计算

### 计算公式

```
任务栈需求（字节）= 局部变量 × sizeof(类型)
                 + 函数调用嵌套深度 × 8~32 字节
                 + 中断栈帧需求（如果有）
                 + 安全余量（20%~30%）
```

### 典型栈消耗估算

| 操作 | 栈消耗（字节） | 说明 |
|------|---------------|------|
| 普通局部变量（int, float） | 4 | 每变量 |
| 结构体局部变量 | sizeof(struct) | |
| 函数调用嵌套（1层） | 32~64 | 返回地址 + 寄存器 |
| 中断嵌套（1层） | 64~96 | Cortex-M 中断栈帧 |
| printf/float 运算 | 200~400 | 很大，慎用 |

### 实际案例计算

```c
void my_task(void *param) {
    // 局部变量
    uint8_t  buffer[64];     // +64 字节
    uint32_t flags;          // +4 字节
    float    temp, humidity; // +8 字节

    // 函数调用（假设嵌套3层）
    // my_task -> process_data -> format_output -> memcpy
    // 每层约 32 字节
    // 3层 × 32 = 96 字节

    // 中断中用到（假设最深1层）
    // +96 字节

    // 额外安全余量 30%
    // 总量 = 64 + 4 + 8 + 96 + 96 = 268
    // 268 × 1.3 = 348 → 取整 384（字）
    // 因为单位是字，所以 384 字 = 1536 字节

    // 推荐写法：
    // xTaskCreate(my_task, "MyTask", 384, NULL, 1, NULL);
}
```

### configMINIMAL_STACK_SIZE

```c
// FreeRTOSConfig.h
#define configMINIMAL_STACK_SIZE ((uint16_t)128)
```

这是 **Idle 任务**和**栈溢出检测钩子**的栈大小。

| 芯片 | configMINIMAL_STACK_SIZE 实际大小 |
|------|----------------------------------|
| Cortex-M0/M0+ | 128 × 4 = 512 字节 |
| Cortex-M3/M4/M7 | 128 × 4 = 512 字节 |
| 某些 Demo 板 | 可能更大（如 256 或 512） |

**如果启用了 `configCHECK_FOR_STACK_OVERFLOW`**，必须确保 `configMINIMAL_STACK_SIZE` 足够大。

## 常见栈配置参考

| 任务类型 | 推荐栈大小（字） | 说明 |
|---------|-----------------|------|
| Idle 任务 | configMINIMAL_STACK_SIZE | 只做最小工作 |
| 简单传感器读取 | 128~192 | 无 printf，无复杂调用 |
| 简单显示更新 | 192~256 | 少量字符串处理 |
| 通信任务（UART/SPI） | 256~384 | 中等缓冲区 |
| 网络协议栈任务 | 512~1024 | 协议栈复杂，缓冲区大 |
| GUI 任务 | 512~2048 | 取决于库 |
| printf 重度用户 | +200~400 | 如果必须用，控制使用频率 |

## 栈溢出的实际案例

### 案例 1：printf 导致栈溢出

```c
void logging_task(void *param) {
    while (1) {
        // ❌ 大字符串栈上分配
        char msg[256];
        snprintf(msg, sizeof(msg), "Sensor: %f, %f", temp, humi);
        send_log(msg);

        // ✅ 推荐：使用静态/全局
        static char msg2[256];
        snprintf(msg2, sizeof(msg2), ...);
    }
}
```

### 案例 2：递归调用栈溢出

```c
void process_data(void *param) {
    uint8_t data[128];

    if (condition) {
        process_data(param);  // ⚠️ 递归调用，每层 +128 字节
    }
}
```

### 案例 3：中断嵌套栈需求

```c
void UART_IRQHandler(void) {
    // 中断栈帧约 64 字节
    // 如果此中断调用了某个任务，该任务的栈帧也叠加
    // 最终：中断栈帧 + 被调用函数栈帧
}
```

## 优先级怎么设

### 基础规则

| 规则 | 说明 |
|------|------|
| **数字越大优先级越高** | configMAX_PRIORITIES = 5 → 优先级 0~4 |
| **高优先级先运行** | 抢占式调度下，高优先级就绪则立即抢占 |
| **留一个优先级给 Idle** | 建议优先级 0 留给 Idle 任务 |
| **优先级反转** | 共享资源时注意（见第 5 章） |

### 优先级分配建议

| 优先级 | 典型任务 | 说明 |
|--------|---------|------|
| 0（最低） | Idle 任务 | 系统自动创建 |
| 1 | 后台显示/日志 | 非关键，可被任何任务抢占 |
| 2 | 传感器读取 | 周期性，适中响应 |
| 3 | 通信任务 | UART/SPI/I2C |
| 4（最高） | 紧急处理/安全 | 看门狗、外部中断触发 |

### configMAX_PRIORITIES 设多少

| 场景 | 推荐值 | 理由 |
|------|--------|------|
| 简单系统（< 5 个任务） | 5 | 够用即可 |
| 中等系统（5~10 个任务） | 8~10 | 每优先级一个任务，留余量 |
| 复杂系统 | 16~32 | 但注意 RAM 消耗（就绪队列数组大小 = 优先级数） |

**注意**：`configMAX_PRIORITIES` 越大，RAM 消耗越多（每个优先级一个就绪队列）。

## 优先级设定常见错误

### 错误 1：所有任务同一优先级

```c
// ❌ 所有任务优先级相同，无法抢占
xTaskCreate(task1, "T1", 256, NULL, 1, NULL);
xTaskCreate(task2, "T2", 256, NULL, 1, NULL);
xTaskCreate(task3, "T3", 256, NULL, 1, NULL);
```

**结果**：同优先级任务按时间片轮转，但如果某个任务阻塞，整个调度会退化。

### 错误 2：高优先级任务不阻塞

```c
// ❌ 高优先级任务死循环，导致低优先级永远无法运行
void high_priority_task(void *param) {
    while (1) {
        do_critical_work();  // 没有 vTaskDelay
        // 永远占着 CPU！
    }
}
```

**死锁场景**：两个任务互相等待对方。

### 错误 3：优先级设反（最低数字最高）

有些 RTOS 习惯把 0 设为最高，FreeRTOS 是**数字越大越高**。如果习惯了其他 RTOS，小心这个坑。

---

## 本节小结

- **栈大小计算**：局部变量 + 调用嵌套 + 中断 + 安全余量，周期性任务建议 `vTaskDelay` 释放 CPU
- **优先级**：数字越大越高，0 留 Idle，至少留 1-2 个最高优先级给安全关键任务
- **configMAX_PRIORITIES**：够用就行，越大 RAM 消耗越多
- **调试栈问题**：启用 `configCHECK_FOR_STACK_OVERFLOW`，观察 `vApplicationStackOverflowHook`
