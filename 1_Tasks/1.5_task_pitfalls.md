# 1.5 踩坑实录：栈溢出怎么排查

## 栈溢出的典型表现

| 现象 | 原因 |
|------|------|
| 系统随机死机 | 栈数据被破坏，TCB 结构损坏 |
| hardfault 频发 | 栈指针越界访问到非法地址 |
| 任务偶发消失 | 被检测到溢出后任务被删除 |
| printf 输出乱码 | 栈上的字符串被覆盖 |
| 功能正常但随机重启 | 看门狗复位 |

---

## 配置栈溢出检测

### Step 1：启用检测

```c
// FreeRTOSConfig.h
#define configCHECK_FOR_STACK_OVERFLOW     2     // 0=关闭, 1=快速检测, 2=完整检测
#define configUSE_MALLOC_FAILED_HOOK      1     // 内存分配失败钩子
```

### Step 2：实现钩子函数

```c
// 必须提供此函数（即使什么都不做）
void vApplicationStackOverflowHook(TaskHandle_t xTask, char *pcTaskName) {
    /* 记录日志或触发看门狗 */
    printf("Stack overflow detected!\r\n");
    printf("Task: %s\r\n", pcTaskName);

    /* 可选：进入临界态，死循环等待调试 */
    while (1) {
        taskENTER_CRITICAL();
    }
}

void vApplicationMallocFailedHook(void) {
    printf("Malloc failed! Increase configTOTAL_HEAP_SIZE\r\n");
    while (1);
}
```

## 排查方法

### 方法 1：开启栈水印检测（推荐）

```c
// 任务执行中获取实际使用的栈大小
void check_task_stack(void) {
    TaskHandle_t task = xTaskGetHandle("MyTask");
    UBaseType_t watermark = uxTaskGetStackHighWaterMark(task);
    printf("Stack high water mark: %u words\r\n", watermark);
}
```

**水线（High Water Mark）** = 历史上栈剩余的最小值，即"最危险时刻用了多少栈"。

### 方法 2：Fill 内存法（调试时）

```c
void fill_stack_for_debug(TaskHandle_t task) {
    StackType_t *stack = get_task_stack(task);
    uint32_t stack_size = get_task_stack_size(task);

    /* 用已知模式填充栈 */
    for (int i = 0; i < stack_size; i++) {
        stack[i] = 0xA5A5A5A;
    }

    /* 运行一段时间后检查 */
    int used_count = 0;
    for (int i = 0; i < stack_size; i++) {
        if (stack[i] != 0xA5A5A5A) used_count++;
    }
    printf("Used stack: %d / %d words\r\n", used_count, stack_size);
}
```

### 方法 3：观察 hardfault

```c
// HardFault_Handler 中查看 PSP（进程栈指针）
// 如果 PSP 指向非法地址，说明栈溢出
void HardFault_Handler(void) {
    __ASM volatile(
        " MRS     r0, PSP\n"
        " CMP     r0, #0\n"
        " BEQ     done\n"
        /* 打印 r0-r12, PC, xPSR 等寄存器 */
        " done:\n"
        " B       .\n"
    );
}
```

## 真实踩坑案例

### 案例 1：printf 吃栈

**现象**：`display_task` 运行几分钟后死机。

**排查**：

```c
void display_task(void *param) {
    while (1) {
        // ❌ printf 吃约 400 字节栈
        printf("Value: %d, %d, %d, %d\r\n", a, b, c, d);

        // ✅ 用串口直接发送
        char buf[32];
        snprintf(buf, sizeof(buf), "Value: %d\r\n", a);
        HAL_UART_Transmit(&huart1, (uint8_t *)buf, strlen(buf), 100);
    }
}
```

**教训**：嵌入式中尽量避免 printf，或重定向到串口发送（不占栈）。

---

### 案例 2：中断里调用任务函数

**现象**：中断触发时系统复位，无规律。

**排查**：

```c
// ❌ 错误：中断里调用了会分配栈的函数
void TIM2_IRQHandler(void) {
    if (tim_flag) {
        process_sensor_data();  // 内部可能用了几百字节栈
        // 中断栈帧 + process_sensor_data 栈帧 > 任务栈上限
    }
}
```

**教训**：中断处理函数要尽可能小，不能调用复杂的任务层函数。

---

### 案例 3：递归死循环

**现象**：某个任务运行一段时间后栈溢出。

**排查**：

```c
// ❌ 递归没终止条件
uint32_t factorial(uint32_t n) {
    return n * factorial(n - 1);  // 栈每层 + 约 32 字节
    // n=10 → 10 × 32 = 320 字节
    // n=20 → 20 × 32 = 640 字节
    // n 无终止 → 栈耗尽
}
```

**教训**：FreeRTOS 任务中严禁递归调用。改用循环实现。

---

### 案例 4：栈空间被其他任务踩

**现象**：`task_A` 莫名其妙数据被改。

**原因**：两个任务栈相邻，task_B 溢出的数据写入了 task_A 的栈。

**教训**：
1. 确保每个任务栈分配足够
2. 开启 `configCHECK_FOR_STACK_OVERFLOW`
3. 运行时监控 `uxTaskGetStackHighWaterMark()`

---

## 栈配置检查清单

```
□ 估算任务最大栈需求（局部变量 + 调用深度 + 中断）
□ 乘以 1.2~1.3 安全系数
□ 转换为 word 单位
□ 确认 pxStack[ulStackDepth] 不会越界
□ 启用 configCHECK_FOR_STACK_OVERFLOW
□ 实现 vApplicationStackOverflowHook
□ 运行时定期检查 uxTaskGetStackHighWaterMark
□ 测试边界条件（最深调用、最大数据处理）
```

---

## 本节小结

- 栈溢出表现多样（死机、hardfault、乱码），排查困难，要提前预防
- `configCHECK_FOR_STACK_OVERFLOW = 2` + 钩子函数是标准配置
- `uxTaskGetStackHighWaterMark()` 是运行时监控的利器
- printf、递归、中断嵌套是栈溢出的三大杀手
