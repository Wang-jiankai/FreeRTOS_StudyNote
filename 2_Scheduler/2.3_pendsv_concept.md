# 2.3 概念：PendSV 中断如何触发上下文切换

## 为什么用 PendSV

ARM Cortex-M 使用 **PendSV（可挂起的系统调用）** 来触发上下文切换，而不是直接切换。

### 为什么不能直接切换

| 问题 | 原因 |
|------|------|
| 需要保存寄存器 | 切换任务要保存所有寄存器 |
| 需要特权模式 | 用户任务在 PSP 栈，特权代码才能操作 |
| 可能在中断中 | 如果在中断中触发，不能影响当前中断 |

**解决方案**：PendSV 是一个**中断**，可以在任何优先级触发（设为最低），等到所有重要中断处理完才执行切换。

---

## Cortex-M 中断优先级

Cortex-M 有 3 个系统异常（固定优先级）：

| 异常 | 优先级 | 用途 |
|------|--------|------|
| **Reset** | -3（最高） | 复位 |
| **NMI** | -2 | 不可屏蔽中断 |
| **HardFault** | -1 | 硬件错误 |

其他可配置优先级异常：

| 异常 | 优先级 | 用途 |
|------|--------|------|
| **SysTick** | 可配置（通常最低） | 系统时钟节拍 |
| **PendSV** | 可配置（通常最低） | 触发上下文切换 |

### 代码流程

```c
// SysTick_Handler (高优先级)
void SysTick_Handler(void) {
    xTaskIncrementTick();  // 查找到期任务，标记需要切换
    portYIELD();          // 触发 PendSV
}

// portYIELD 实际做的事
#define portYIELD() \
    SCB->ICSR |= SCB_ICSR_PENDSVSET_Msk;  // 设置 PendSV 请求位

// 等 SysTick 退出后，PendSV 开始执行

// PendSV_Handler (低优先级，最后执行)
void PendSV_Handler(void) {
    /* 1. 保存当前任务寄存器到 PSP 栈 */
    __asm volatile(
        " MRS    r0, PSP             \n"
        " STMDB  r0!, {r4-r11}      \n"  // 保存 r4-11
        " BL     vTaskSwitchContext  \n"  // 选择下一个任务
        /* 2. 恢复新任务寄存器 */
        " LDMIA  r0!, {r4-r11}      \n"  // 恢复 r4-11
        " MSR    PSP, r0             \n"  // 更新 PSP
        " BX     lr                  \n"  // 返回
    );
}
```

## 为什么 Cortex-M 要这样设计

| 设计 | 原因 |
|------|------|
| 用中断实现切换 | 需要特权级才能操作 PSP/切换栈 |
| PendSV 最低优先级 | 确保切换不会打断任何实时响应 |
| 分离触发和执行 | Tick 中断标记需要切换，PendSV 统一执行 |

---

## 本节小结

- **PendSV** = 可挂起的系统调用，是 Cortex-M 实现上下文切换的机制
- 触发方式：向 `ICSR[28]` 写 1（`portYIELD()`）
- **PendSV 设为最低优先级**，确保所有重要中断处理完才切换
- 典型流程：Tick 中断标记切换 → 退出所有中断 → PendSV 执行实际切换
- 切换本质：保存当前任务寄存器到栈 → 更新 `pxCurrentTCB` → 恢复新任务栈帧
