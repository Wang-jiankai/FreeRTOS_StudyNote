# 2.5 踩坑实录：调度器不启动 / 优先级反转

## 坑 1：调度器启动失败

### 现象

`vTaskStartScheduler()` 调用后系统卡死或复位。

### 常见原因

#### 原因 1：内存不足

```c
/* vTaskStartScheduler 内部会创建 Idle 任务和 Timer daemon */
BaseType_t xTaskStartScheduler(void) {
    /* 创建 Idle 任务（如果 configUSE_IDLE_HOOK = 1） */
    /* 创建 Timer daemon（如果 configUSE_TIMERS = 1） */

    /* 如果内存不足，返回 err */
    if (xReturn != pdPASS) {
        return xReturn;  // 系统启动失败
    }
}
```

**排查**：

```c
// FreeRTOSConfig.h
#define configTOTAL_HEAP_SIZE    ((size_t)10240)  // 太小？

// 检查：所有任务的栈 + Idle 栈 + Timer 栈 + 内核对象 < configTOTAL_HEAP_SIZE
```

#### 原因 2：Tick 中断未配置

```c
// FreeRTOS 依赖 SysTick 或其他定时器提供 Tick 中断
// 如果你没配置，调度器不会运行

// 检查 port.c 中是否有 Tick 配置：
void xPortStartScheduler(void) {
    /* 配置 SysTick */
    portNop();
    // 如果这里卡住，说明 Tick 没正确配置
}
```

#### 原因 3：堆栈对齐问题

ARM Cortex-M 要求 8 字节对齐。

```c
// 如果 heap_4.c 的分配不满足对齐要求
// 可能导致 hardfault

// 确保：
// #define portBYTE_ALIGNMENT 8
```

## 坑 2：任务永远不运行（调度器好像没工作）

### 现象

任务创建成功，但永远不执行。

### 常见原因

#### 原因 1：所有任务都阻塞了

```c
// ❌ 所有任务都延时很长，系统空转 Idle 任务
void task1(void *p) { vTaskDelay(pdMS_TO_TICKS(60000)); } // 60秒
void task2(void *p) { vTaskDelay(pdMS_TO_TICKS(60000)); } // 60秒
void task3(void *p) { vTaskDelay(pdMS_TO_TICKS(60000)); } // 60秒
```

**排查**：减少延时时间，确认任务能周期性就绪。

#### 原因 2：优先级设错

```c
// ❌ 所有任务优先级相同，且都在阻塞
// 高优先级任务在阻塞队列，不唤醒低优先级
```

#### 原因 3：PendSV 未使能

```c
// 检查 startup_*.s 中是否正确使能了 PendSV
// 应该 NVIC_EnableIRQ(PendSV_IRQn);
```

## 坑 3：优先级反转

### 概念

高优先级任务等待低优先级任务持有的资源，导致实际执行顺序反转。

```
[IMAGE_PROMPT: Priority inversion diagram showing 3 tasks: High (H), Medium (M), Low (L). Timeline showing 6 phases:
Phase 1: L runs and acquires mutex
Phase 2: L blocks (I/O or delay)
Phase 3: H tries to acquire mutex but blocked by L
Phase 4: M preempts L and runs (M > L but M doesn't need the mutex!)
Phase 5: M finishes
Phase 6: L runs again, releases mutex, H finally gets mutex and runs
Total time: T_actual >> T_expected
Label "优先级反转" and show "H 被 M 间接延迟" with arrows. Use Chinese labels throughout.}
```

### 真实案例

```c
// 场景：GPS 任务（高优先级）等待互斥量，但互斥量被 WiFi 任务（低优先级）持有

void gps_task(void *p) {          // 高优先级
    while (1) {
        xSemaphoreTake(gps_mutex, portMAX_DELAY);  // 等低优先级释放
        read_gps_data();
        xSemaphoreGive(gps_mutex);
        vTaskDelay(pdMS_TO_TICKS(100));
    }
}

void wifi_task(void *p) {          // 低优先级
    while (1) {
        xSemaphoreTake(gps_mutex, portMAX_DELAY);  // 先持有
        send_wifi_data();  // 耗时
        xSemaphoreGive(gps_mutex);
        vTaskDelay(pdMS_TO_TICKS(1000));
    }
}

void display_task(void *p) {        // 中优先级
    while (1) {
        // 这个任务在 wifi 和 gps 之间插入
        update_display();
        vTaskDelay(pdMS_TO_TICKS(10));
    }
}
```

**问题**：当 WiFi 任务持有互斥量时，GPS 任务阻塞。此时 Display 任务（中等优先级）抢占 WiFi 任务，导致 GPS 延迟。

## 坑 4：死锁（Deadlock）

### 概念

两个或多个任务互相等待对方持有的资源。

```c
// ❌ 死锁示例
void task1(void *p) {
    while (1) {
        xSemaphoreTake(mutex1, portMAX_DELAY);   // 拿 mutex1
        vTaskDelay(pdMS_TO_TICKS(1));             // 假装做点事
        xSemaphoreTake(mutex2, portMAX_DELAY);   // 等 mutex2
        // ...
        xSemaphoreGive(mutex2);
        xSemaphoreGive(mutex1);
    }
}

void task2(void *p) {
    while (1) {
        xSemaphoreTake(mutex2, portMAX_DELAY);   // 拿 mutex2
        vTaskDelay(pdMS_TO_TICKS(1));
        xSemaphoreTake(mutex1, portMAX_DELAY);   // 等 mutex1
        // ...
        xSemaphoreGive(mutex1);
        xSemaphoreGive(mutex2);
    }
}
```

```
[IMAGE_PROMPT: Deadlock diagram showing two tasks and two mutexes. Task1 holds Mutex1 and waits for Mutex2. Task2 holds Mutex2 and waits for Mutex1. Show circular wait arrow between them. Label "死锁：互相等待对方持有的资源" in Chinese. Show time axis indicating deadlock state persists forever.}
```

### 预防死锁

| 方法 | 说明 |
|------|------|
| **固定顺序获取锁** | 所有任务按相同顺序获取多个锁 |
| **timeout** | 获取锁加超时，不要无限等待 |
| **检测重启** | 计数超时次数，超限复位 |

```c
// ✅ 修复：按固定顺序
void task1(void *p) {
    xSemaphoreTake(mutex1, portMAX_DELAY);
    xSemaphoreTake(mutex2, portMAX_DELAY);
    // ...
}

void task2(void *p) {
    xSemaphoreTake(mutex1, portMAX_DELAY);  // 顺序和 task1 一致
    xSemaphoreTake(mutex2, portMAX_DELAY);
    // ...
}
```

## 坑 5：调度器被意外挂起

### 概念

调用了 `vTaskSuspendAll()` 但没对应 `xTaskResumeAll()`。

```c
// ❌ 调度器被永久挂起
void bad_task(void *p) {
    vTaskSuspendAll();  // 挂起
    if (some_error) {
        return;  // ❌ 提前返回，没调用 xTaskResumeAll
    }
    xTaskResumeAll();
}
```

**注意**：`vTaskSuspendAll()` 可以嵌套，嵌套次数要匹配。

## 坑 6：时间片不工作

### 现象

同优先级任务应该轮转，但只有一个任务在运行。

### 原因

```c
// FreeRTOSConfig.h
#define configUSE_TIME_SLICING    0   // ❌ 时间片被关闭
```

或者：

```c
// 如果任务主动调用了 vTaskDelay，会进入阻塞态
// 不在就绪队列，不会参与时间片轮转
void task(void *p) {
    while (1) {
        do_work();
        vTaskDelay(pdMS_TO_TICKS(10));  // ⚠️ 延时期间不参与时间片
    }
}
```

## 排查清单

```
□ vTaskStartScheduler 返回值是什么？errCOULD_NOT_ALLOCATE_REQUIRED_MEMORY？
□ configTOTAL_HEAP_SIZE 是否足够？（所有任务栈 + 内核对象）
□ 是否有任务在创建后立即进入阻塞？（vTaskDelay 在创建后立即调用）
□ Tick 中断是否正常？（用 GPIO 翻转 + 示波器验证）
□ PendSV 中断是否使能？（NVIC）
□ 是否发生死锁？（两个任务互相等待）
□ 是否发生优先级反转？（用互斥量代替信号量解决）
□ configUSE_TIME_SLICING 是否为 1？
```

---

## 本节小结

- 调度器启动失败：优先排查内存不足和 Tick 中断配置
- 任务不运行：检查所有任务是否都阻塞，或 PendSV 未使能
- **优先级反转**：用互斥量代替信号量，或启用优先级继承
- **死锁**：固定锁获取顺序，加 timeout
- 调度器挂起/恢复必须配对，注意提前返回路径
