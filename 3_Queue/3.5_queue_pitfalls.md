# 3.5 踩坑实录：队列死锁 / 超时设置

## 坑 1：队列死锁

### 现象

两个任务互相等待对方通过队列发送数据，永远卡死。

```c
// ❌ 死锁示例
void task_A(void *p) {
    uint32_t data;
    while (1) {
        // task_A 等待 task_B 发数据
        xQueueReceive(xQueue_AB, &data, portMAX_DELAY);  // 等

        // task_A 处理完，想把结果发回 task_B
        xQueueSend(xQueue_BA, &data, portMAX_DELAY);      // 等
    }
}

void task_B(void *p) {
    uint32_t data;
    while (1) {
        // task_B 先发送，再等待结果
        data = 42;
        xQueueSend(xQueue_AB, &data, portMAX_DELAY);     // 发到 AB

        // 等待 task_A 的结果
        xQueueReceive(xQueue_BA, &data, portMAX_DELAY);  // 等
    }
}
```

**问题**：task_A 在 `xQueueReceive(xQueue_AB)` 等待，task_B 在 `xQueueReceive(xQueue_BA)` 等待，互相死等。

```

Circular arrow with label "死锁" in Chinese. Below show deadlock state persists forever without intervention.}
```

### 解决方案

| 方法 | 说明 |
|------|------|
| **单向数据流** | 一个队列解决，不要互相等待 |
| **加超时** | `xTicksToWait` 用有限时间而非 `portMAX_DELAY` |
| **改变设计** | 生产者-消费者模型，单向传递 |

```c
// ✅ 加超时，并处理超时情况
if (xQueueReceive(xQueue_AB, &data, pdMS_TO_TICKS(100)) == pdTRUE) {
    // 收到数据
} else {
    // 超时，可能是死锁或对方异常
}
```

## 坑 2：队列满导致发送任务永久阻塞

### 现象

生产者速度 > 消费者速度，队列满后生产者永远卡死。

```c
void producer_task(void *p) {
    uint32_t count = 0;
    while (1) {
        // ❌ 队列深度 10，生产速度 > 消费速度
        // 队列满后这里永久阻塞
        xQueueSend(xQueue, &count, portMAX_DELAY);  // 等
        count++;
    }
}
```

### 排查方法

```c
// 检查队列已满和发送失败
UBaseType_t spaces = uxQueueSpacesAvailable(xQueue);
if (spaces == 0) {
    // 队列满了
    // 发送失败的风险
}
```

### 解决方案

| 方法 | 说明 |
|------|------|
| **加超时** | 用有限超时，而非无限等待 |
| **背压机制** | 生产者检测队列可用空间再发送 |
| **改队列深度** | 增加队列深度 |
| **流控** | 消费者主动通知生产者暂停 |

```c
// ✅ 加超时
if (xQueueSend(xQueue, &count, pdMS_TO_TICKS(10)) != pdTRUE) {
    // 队列满，跳过本次发送
    count++;  // 计数器仍增加
}
```

## 坑 3：接收超时设置不合理

### 现象

任务用 `portMAX_DELAY` 永久等待，但数据已经不可能再来了。

```c
// ❌ 不合理：假设串口断线后，队列永不再有数据
void uart_task(void *p) {
    uint8_t ch;
    while (1) {
        xQueueReceive(xUARTQueue, &ch, portMAX_DELAY);  // ❌ 永久等待
        process_uart_data(ch);
    }
}
```

### 场景分析

| 场景 | 推荐超时 |
|------|---------|
| 生产者持续运行 | `portMAX_DELAY` 可接受 |
| 生产者可能异常退出 | 有限超时 |
| 中断触发 | `0`（立即返回） |
| 轮询设备 | 有限超时 |

```c
// ✅ 合理：假设 1 秒没有数据则认为异常
TickType_t timeout = pdMS_TO_TICKS(1000);
while (1) {
    if (xQueueReceive(xUARTQueue, &ch, timeout) == pdTRUE) {
        process_uart_data(ch);
    } else {
        // 超时，可能是断线
        reconnect_uart();
    }
}
```

## 坑 4：结构体传递问题

### 现象

发送结构体后，接收到的数据不对。

```c
typedef struct {
    uint8_t id;
    float value;
} SensorData;

// ❌ 创建队列时元素大小传错了
xQueue = xQueueCreate(10, sizeof(float));  // 只传了 float 大小
//                              ^^^^^^^^^^^^
//                              应该是 sizeof(SensorData)

// 发送
SensorData data = {1, 3.14f};
xQueueSend(xQueue, &data, portMAX_DELAY);  // ⚠️ 拷贝了 5 字节（id + value）
// 但队列只能存 4 字节（float 大小）
```

### 正确做法

```c
// ✅ 正确：明确用 sizeof
xQueue = xQueueCreate(10, sizeof(SensorData));  // 5 字节 × 10 = 50 字节

// ✅ 更安全：运行时检查
configASSERT(10 * sizeof(SensorData) <= configTOTAL_HEAP_SIZE);
```

## 坑 5：中断优先级导致 FromISR 不工作

### 现象

中断里调用 `xQueueSendFromISR`，但数据永远发不出去，任务也收不到。

### 原因

中断优先级 ≥ `configMAX_SYSCALL_INTERRUPT_PRIORITY`，被 FreeRTOS 禁止调用 API。

```c
// FreeRTOSConfig.h
#define configMAX_SYSCALL_INTERRUPT_PRIORITY  5

// 某中断配置
NVIC_SetPriority(UART1_IRQn, 3);  // ❌ 优先级 3 < 5，禁止调用 API

// 在这个中断里调用任何 FromISR 都会被忽略
```

### 排查

```c
// 调试：打印 cIsLocked
// 如果 cIsLocked > 0，说明队列被锁
```

### 解决

```c
// ✅ 正确：优先级数值要 > configMAX_SYSCALL_INTERRUPT_PRIORITY
// 数值越大，优先级越低
NVIC_SetPriority(UART1_IRQn, 6);  // ✅ 优先级 6 > 5，可以调用
```

## 坑 6：往已删除队列发送

### 现象

任务 A 删除了队列，任务 B 还在往里发，但数据丢失。

```c
// ❌ 示例
void task_B(void *p) {
    while (1) {
        // task_A 已经删除了 xQueue，但 task_B 不知道
        xQueueSend(xQueue, &data, portMAX_DELAY);  // 数据丢失！
    }
}
```

**排查**：删除队列前，确保所有等待队列的任务都被唤醒。

```c
// ✅ 安全删除
vQueueDelete(xQueue);  // 不会唤醒等待中的任务

// ✅ 推荐：先用 vTaskSuspendAll 暂停所有任务，再删除
```

## 队列调试清单

```
□ 队列创建是否成功？（检查返回值 NULL）
□ 队列深度和元素大小是否正确？（sizeof 计算）
□ 超时是否合理？（portMAX_DELAY 用于永久等待场景）
□ 中断优先级是否 < configMAX_SYSCALL_INTERRUPT_PRIORITY？
□ FromISR 是否传入 &pxHigherPriorityTaskWoken 并判断？
□ 是否处理了 errQUEUE_FULL 情况？
□ 是否有死锁风险？（任务间单向数据流最稳）
□ 删除队列前是否已停止所有使用它的任务？
```

## 本节小结

- **死锁**：两个任务互相等待对方发队列数据，加超时或改单向数据流
- **队列满**：生产者快于消费者时发送任务永久阻塞，加超时或流控
- **超时不合理**：`portMAX_DELAY` 只用于生产者是持续运行的场景
- **结构体大小**：`sizeof(SensorData)` 而非单个字段大小
- **中断优先级**：必须确保 < `configMAX_SYSCALL_INTERRUPT_PRIORITY`
- **队列删除**：不唤醒等待任务，删除前确保没有任务在使用
