# 0.1 什么是 RTOS / 为什么需要

## 概念

**RTOS = Real-Time Operating System（实时操作系统）**

关键在于"实时"二字——任务必须在确定的时间内完成响应。

### 硬实时 vs 软实时

| 类型 | 响应超时后果 | 例子 |
|------|-------------|------|
| **硬实时** | 系统失效（灾难性） | 汽车安全气囊、飞控 |
| **软实时** | 体验下降，可接受 | 音视频播放、网络传输 |

FreeRTOS 属于**软实时**，但响应确定性很高，适合大多数嵌入式场景。

---

### 什么是 FreeRTOS

- **诞生**：2003 年，Richard Barry 开源
- **定位**：微控制器专用 RTOS
- **代码量**：非常小（以内核计算，约 9000 行 C 代码）
- **特点**：可配置、跨平台、实时性强

## 为什么需要 RTOS

### 裸机（裸机循环）的问题

```c
int main(void) {
    init_all();
    while (1) {
        read_sensor();      // 假设耗时 10ms
        process_data();     // 假设耗时 5ms
        update_display();   // 假设耗时 20ms
        // 总周期：35ms
        // 如果 read_sensor 变成 100ms 呢？
    }
}
```

**裸机的核心缺陷**：

1. **响应不确定** — 所有任务排队执行，一个慢拖累全部
2. **CPU 空转** — 等待时也在跑 while 循环，功耗高
3. **难以扩展** — 加一个功能就要改整个循环
4. **实时性差** — 中断来了也要等当前代码跑完

### RTOS 如何解决

```
[IMAGE_PROMPT: Comparison diagram showing bare metal loop on left (tasks in series) and RTOS on right (tasks running concurrently with arrows). Show task A, B, C running in parallel with different priorities, with interrupt handlers separate.]
```

```c
int main(void) {
    init_all();
    xTaskCreate(task_sensor, "Sensor", 256, NULL, 2, NULL);
    xTaskCreate(task_display, "Display", 256, NULL, 1, NULL);
    vTaskStartScheduler();  // 启动调度器，永不返回
    while (1);              // 不会执行到这里
}
```

## 核心术语速查

| 术语 | 含义 |
|------|------|
| **Task（任务）** | 独立运行的基本单元，类似于"线程" |
| **Scheduler（调度器）** | 决定哪个任务在哪个时刻运行 |
| **ISR（中断服务程序）** | 硬件中断触发的紧急处理 |
| **Context Switch（上下文切换）** | CPU 从一个任务切换到另一个任务 |
| **Priority（优先级）** | 任务的重要程度，调度器据此决定执行顺序 |
| **Blocking（阻塞）** | 任务主动放弃 CPU 等待某个事件 |
| **Preemption（抢占）** | 高优先级任务可以中断低优先级任务 |

---

## 什么时候不用 RTOS

| 场景 | 推荐方案 |
|------|----------|
| 简单控制逻辑，任务 < 3 个 | 裸机状态机 |
| 资源极其紧张（4KB RAM 以下） | 裸机 |
| 硬件不支持中断嵌套 | 裸机或简化 RTOS |
| 确定性要求极高（< 1μs） | 裸机或裸机 + 中断 |

---

## 本节小结

- RTOS 的核心价值是**确定性调度**和**并发执行**
- FreeRTOS 定位**微控制器**，代码量小、配置灵活
- 适合多任务、事件驱动、需要一定实时性的嵌入式场景
- 不是所有项目都需要 RTOS，简单场景裸机更合适
