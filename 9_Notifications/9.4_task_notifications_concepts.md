# 9.4 概念：任务通知（Task Notifications）

## 为什么需要任务通知

传统的任务间同步需要创建中间对象（队列、信号量、事件组）：

```c
// 传统方式：需要先创建信号量
SemaphoreHandle_t xSem = xSemaphoreCreateBinary();
xSemaphoreTake(xSem, portMAX_DELAY);  // 任务等待
xSemaphoreGiveFromISR(xSem);           // ISR 发信号
```

任务通知**不需要任何中间对象**，直接通知目标任务：

```c
// 任务通知：不需要创建任何对象
xTaskNotifyGive(xTaskToNotify);         // ISR 直接通知目标任务
ulTaskNotifyTake(pdTRUE, portMAX_DELAY); // 任务等待
```

## 任务通知的本质

每个任务都有一个**内置的通知值**（TCB 中的数组）：

```c
// TCB 中的通知字段（v10.4+ 支持数组）
typedef struct tskTaskControlBlock {
    #if (configUSE_TASK_NOTIFICATIONS == 1)
        volatile uint32_t ulNotifiedValue[configTASK_NOTIFICATION_ARRAY_ENTRIES];
        volatile uint8_t ucNotifyState[configTASK_NOTIFICATION_ARRAY_ENTRIES];
    #endif
} tskTCB;
```

**通知值**是一个 `uint32_t`，可以：
- **递增**（像二进制/计数信号量）
- **写入新值**（覆盖）
- **按位 OR**

---

## 通知状态

每个通知槽有独立的状态：

```c
typedef enum {
    taskNOT_WAITING_NOTIFICATION = 0,  /* 未等待 */
    taskWAITING_NOTIFICATION,             /* 等待中 */
} eNotifyState;
```

任务调用 `ulTaskNotifyTake()` 时，状态变为"等待中"；收到通知后状态变为"未等待"。

---

## 与其他方式的对比

| 特性 | 任务通知 | 信号量 | 队列 |
|------|---------|--------|------|
| 是否需要创建对象 | ❌ 不需要 | ✅ 需要 | ✅ 需要 |
| 是否能传数据 | ✅ 32 位值 | ❌ 不能 | ✅ 可以 |
| ISR 通知任务 | ✅ 直接 | ✅ FromISR | ✅ FromISR |
| 多任务广播 | ❌ 只能指定一个任务 | ❌ | ✅ |
| 内存占用 | 几乎为 0 | 几十字节 | 几十到几百字节 |

---

## 使用场景

### 最佳场景

1. **ISR 快速通知任务**（替代二进制信号量，最省内存）
2. **任务间单向同步**（一个任务通知，另一个等待）
3. **计数器**（频繁递增的场景）

### 不适合的场景

1. 需要传递大量数据 → 用队列
2. 需要多个任务等待同一个事件 → 用事件组
3. 需要超时/等待多个条件 → 用信号量或队列

---

## 本节小结

- **任务通知** = 每个任务内置的通知机制，不需要创建信号量/队列
- 本质是 TCB 中的 `ulNotifiedValue[]` 和 `ucNotifyState[]`
- ISR 可以直接通知目标任务，无需中间对象
- 适合"轻量级单向通知"场景，不适合"传递数据"或"多任务广播"
